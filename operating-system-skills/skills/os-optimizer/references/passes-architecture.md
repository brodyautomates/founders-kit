# F9 — Architecture & Discoverability (pass implementation)

**Reference (the why):** `references/anthropic-architecture.md`.
**Applies to:** the whole vault — root `CLAUDE.md`, **every folder regardless of role layer** (curated, session, archive, custom, unknown), every folder-index file (whatever the discovered convention is), every `.md` reachable from the routing chain. Reads `roles.identity` and `roles.context` (whichever the user has) to understand their world.

**Custom roles are full participants in F9.** A custom `building` role is checked for: routing-table coverage (F9.1), folder-index presence (F9.2), discoverability (F9.3), semantic alignment with its inferred purpose (F9.4), duplication against other folders (F9.5), and inclusion in reorg proposals (F9.6) and orientation answers (F9.7). The only filter F9 applies is layer-based exclusion for archive/meta in *some* checks (for example F9.3 excludes archive from discoverability orphan reporting, because archived files are not supposed to be reachable).

**Out of scope (F9 never edits without per-item user approval):** any CLAUDE.md, any SKILL.md, any `.claude/rules/` file. These can be *targeted* by F9 fixes (for example F9.1 rewrites routing, F9.7 adds an orientation section), but only through walk-and-confirm, never automated.

## How this pass works

Whole-vault structural reasoning. The triggers are mechanical (does the path exist? does the folder-index file exist under the discovered convention? does file Y appear in the index's children?), but every finding requires the agent to read folder contents, judge alignment with stated intent, and propose a concrete change. Nothing in F9 ships as flag-only.

## The folder-index convention is *discovered*, not imposed

Step 1.5 detects the file name the user already uses for folder indexes: it could be `README.md`, `Plot.md`, `index.md`, `_index.md`, or a folder-level `CLAUDE.md`. F9.2 enforces *that* convention; it never imposes a specific name. If no convention exists yet (`roles.folder_index_convention` is missing or low-confidence), F9.0 surfaces the gap and proposes adopting one. The user picks the name during the walk; the default suggestion is `README.md` because it is universally understood.

The agent-drafted folder-index file (whatever its name) follows this format:

```markdown
---
status: active
type: folder-index
tags: [folder-index]
date: 2026-05-08
---

# {Folder name}

**Purpose:** {1-line statement of what this folder holds and why}

## Children

- [[file-name]] — {1-line description}
- [[subfolder/{convention-name}]] — {1-line subfolder purpose}
- …

## Updated

2026-05-08 — auto-generated by os-optimizer F9.2.
```

Constraints: ≤ 8KB (well under F5 budget), no em dashes (G7.1), G7.2-compliant frontmatter, markdown-only.

A folder is "non-trivial" if it holds ≥ 2 `.md` files OR ≥ 1 subfolder. Folders with a single file or a single subfolder need no folder-index file, since the file or subfolder is its own index.

## Setup — read identity/context, build navigation graph

Before running any F9 check, do this once and cache it:

1. **Read `roles.identity.path` + `roles.context.path`** — every file under each (if those roles exist). Pull out: who the user is, what the org or business is, the current strategy, the brand voice. F9.7 uses this to build vault-specific orientation questions, and F9.1/F9.4 use it to judge whether a folder description matches the user's actual world. If both roles are missing, F9.7 still runs but draws its orientation questions from generic operator patterns; the absent identity/context is itself an F9.0 finding.
2. **Parse root CLAUDE.md routing table** — extract `{path → description}` pairs. If no routing table exists, that is an F2.6 finding (already handled there); F9.0 surfaces a parallel structural finding about discoverability.
3. **Build the navigation graph:**
   - Node = file or folder.
   - Edge = "reachable from" (root CLAUDE.md → routing entry → folder-index file (under `roles.folder_index_convention.name`) → child file).
   - Compute the hop distance from root for every `.md`.
4. **Build the folder inventory:**
   - For each folder: list of `.md` children, list of subfolders, presence of the discovered folder-index file (whatever its name), mtime of that file if present.
5. **Sample folder contents:**
   - For each folder with ≥ 3 files: pick 3–5 (newest, oldest, median by size). Cache their first 1500 chars plus headers. F9.1 / F9.4 read these to judge alignment.

## F9.0 — Structural-role gaps and clarifications (three-tier)

**Framework rule:** F9.0 surfaces structural needs the agent identifies in *this specific vault*. Three tiers, ranked by how strongly the agent recommends adopting the change:

### Tier 1 — Functional gaps (the optimizer's own job is at risk)

The agent cannot do its job without these. The user has to pick a path forward in the walk.

| Gap | Why it's tier 1 |
|---|---|
| Cannot find first-person operator/identity context anywhere (no folder, no file, no frontmatter section, no inline mention on root CLAUDE.md) | Without this, F8/F9 can't ground judgments in the user's world; reasoning is generic |
| Cannot navigate from root to leaves at all (root CLAUDE.md has no routing of any shape — no table, no prose pointing to folders, no wikilinks) | Discoverability walk has nowhere to start |
| Low-confidence custom roles (`confidence: low` or `unknown`) | Downstream frameworks need to know what layer these participate in |

**Tier 1 is form-agnostic.** Identity satisfied by frontmatter on root CLAUDE.md is fine. Routing satisfied by agent prose (`"the auth code lives in src/server/auth/"`) instead of a table is fine. The folder-index convention is **not** in tier 1: flat or tag-driven vaults do not need it.

**Severity:** fail.

### Tier 2 — Functional improvements (agent judges this would meaningfully help)

These are Brody Operating System conventions the agent has decided would *concretely improve this specific vault* given the current state. Not pushed by default: the agent has to justify each one with reasoning tied to the user's vault. Examples:

- "Adding a `Projects/` (or equivalent) folder would help: you have 14 work-shaped notes scattered across 6 folders with no clear hub, and root CLAUDE.md doesn't route to any of them. Centralizing them would make project status discoverable in one hop."
- "Adopting a folder-index convention would help: 23 of your 31 folders have no index file; the discoverability walk failed for 47 files. Picking a name (README.md by default) and generating indexes resolves all 47 in one pass."
- "Adding a `Decisions/` folder (or equivalent) would help: I found 8 decision-shaped notes in your daily logs — `decided to`, `going with`, `pivoting from` — that are durable knowledge buried in ephemeral logs. A dedicated decisions folder makes them findable."

**Each tier-2 finding is a judgment call**, with reasoning specific to the case. The agent must *not* propose tier-2 findings on the basis of "the Brody Operating System uses this." It proposes them when the current vault state shows the convention would resolve a real problem.

If the user's existing structure already meets the functional need (for example a custom `Lab/` folder is functionally serving as projects), no tier-2 finding fires for `projects`: the function is already met under another name.

**Severity:** warn. Fix proposed by default in the walk; the user can apply (create the folder + starter index, optionally migrate existing files), modify (rename, pick a different parent), or decline (recorded, persists to `.claude/vault-roles.json` so it doesn't re-prompt).

### Tier 3 — Inspiration (the Brody Operating System uses these — adopt any that fit how you work)

For standard roles the agent did not recommend in tier 2 but the user might still want to know about. Surfaced as a single info-level finding listing the Brody Operating System standard taxonomy with one-liners, not as separate findings per missing role. Default decline; the user opts in if any resonate.

**Severity:** info. A fix is offered (adopt one) but defaults to "no thanks."

### Common rules across tiers

- **Already covered by another role (multi-role assignment)** → if the user has a custom folder functionally playing a standard role's part (for example `Knowledge/` covers both context and resources), drop the missing finding and record the merge in the registry as a confirmed multi-role assignment. No tier-2 or tier-3 finding fires for the satisfied function.
- **Persistence** → every "decline" is persisted to `.claude/vault-roles.json` so future runs do not re-prompt the same suggestion.
- **No tier may justify itself with "the Brody Operating System uses this."** Tier 1 cites a functional break; tier 2 cites specific evidence in this vault; tier 3 is openly framed as inspiration the user can take or leave.

**Fixable:** true. Walk-only across all three tiers.

This is what makes the optimizer transferable. The agent does not impose a schema. It observes the user's actual structure, identifies real problems with reasoning grounded in *their* vault, and applies the Brody Operating System taxonomy as a tested-but-optional reference for cases where it would concretely help.

## F9.1 — Routing table truthfulness

**Framework rule:** every routing entry must point to an existing path, and its description must match what is actually inside.

**Trigger heuristic:**
- For each `{path → description}` in the routing table:
  - **Existence check:** does `path` resolve as a file or folder? If not → candidate.
  - **Description check:** read 3–5 sampled files from `path` (cached in setup). Does the routing description align with what the files actually contain? If not → candidate.
  - **Coverage check:** does any top-level folder *not* appear in the routing table? If so → candidate.
  - **Stale entry:** does the routing description reference concepts not in `roles.context.path` (for example a business focus the user has since pivoted away from, per a strategy file under their context role)? → candidate.

**Agent judgment:**
- For a nonexistent path → propose: rewrite the path to the actual location OR remove the entry if the folder was deleted on purpose.
- For a misaligned description → propose: rewrite the description to match folder reality.
- For an uncovered folder → propose: add a routing entry with an agent-drafted description.
- For a stale entry → propose: rewrite to the current strategy per the user's context-role files.

**Fix proposal (every finding has one):** the agent drafts the exact replacement line(s) for the routing table. Walk-only: the user confirms per entry.

**Severity:** fail (a broken existence check breaks the discovery chain). Description misalignment, coverage, and stale entries are warn, but per the optimizer's apply-everything rule, every warn ships a fix.

**Fixable:** true. Walk-only.

## F9.2 — Folder-index presence and freshness

**Framework rule:** every non-trivial folder has an up-to-date folder-index file under the user's discovered convention (`roles.folder_index_convention.name` — could be `README.md`, `{INDEX}`, `index.md`, etc.).

**Pre-condition:** if `roles.folder_index_convention` is missing or low-confidence, F9.0 fires first to establish a convention. Once the user picks one in the walk, F9.2 proceeds using that name. F9.2 never assumes a name on its own.

**Trigger heuristic** (using `INDEX = roles.folder_index_convention.name`):
- For each non-trivial folder (≥ 2 files OR ≥ 1 subfolder):
  - **Missing:** no `{INDEX}` file → candidate.
  - **Stale (children drift):** `{INDEX}` exists but the children list doesn't match `ls` output (files in the folder not in the index, or files in the index no longer in the folder) → candidate.
  - **Stale (mtime):** `{INDEX}` older than the newest file in the folder → candidate.
  - **Wrong format:** `{INDEX}` doesn't follow the format spec above (missing frontmatter, no Purpose line, no Children section) → candidate.

**Agent judgment:**
- For each candidate, read the folder contents. Decide:
  - **Generate from scratch** (missing): the agent drafts the index file per the format spec, with a 1-line description per child inferred from each child's H1 or first paragraph.
  - **Regenerate** (stale): rewrite the Children list; keep the Purpose line if still valid, otherwise rewrite that too.
  - **Repair** (wrong format): convert to spec format, preserving content.

**Fix proposal:** the drafted index file content, ready to write under `{INDEX}`. Walk-only: the user reviews and confirms per folder, and can edit the Purpose line and any child description before applying.

**Severity:** warn. Per the apply-everything rule, every finding ships a fix.

**Fixable:** true. Walk-only.

## F9.3 — Discoverability walk

**Framework rule:** every `.md` file (excluding technical skips and known meta files) must be reachable from root `CLAUDE.md` in ≤ 3 hops via the navigation chain (routing → {INDEX} → file, or routing → file directly).

**Trigger heuristic:**
- Use the navigation graph from setup.
- For each `.md` file in scope:
  - Hop distance > 3 → candidate (deep-buried).
  - Hop distance = ∞ (unreachable) → candidate (navigation orphan).
- Excluded from the check: `CLAUDE.md` itself, `{INDEX}` files (they are navigation nodes, not destinations), `MEMORY.md`, files in `.claude/`, `.git/`, etc. (technical skips), files inside `roles.archive.path` (intentionally not navigated).

**Agent judgment:**
- For each unreachable file, read it and its containing folder's {INDEX} (if any). Decide:
  - **Add to {INDEX}** (most common): the file sits in a folder whose {INDEX} doesn't list it. Propose adding the entry to {INDEX}.
  - **Add routing entry** (folder unrouted): the folder has no routing entry; propose adding one.
  - **Move to discoverable folder** (file misplaced): the folder is intentionally untracked; propose relocating the file.
  - **Archive** (file is dead): the file is genuinely obsolete; propose moving it to `roles.archive.path` (or propose creating an archive role via F9.0 if one is missing).

**Fix proposal:** for each navigation orphan, the agent picks the most likely repair (add-to-Plot, add routing, move, archive) and presents that as the default. Walk-only: the user can pick a different repair per item.

**Severity:** fail. Discoverability is the load-bearing property.

**Fixable:** true. Walk-only.

## F9.4 — Semantic path allocation

**Framework rule:** every file lives in a folder whose stated purpose matches the file's content. **Folder purpose comes from the registry entry's `purpose` field plus the `{INDEX}` Purpose line — both standard and custom roles participate.**

**Trigger heuristic:**
- For each `.md` file: read its parent folder's registry entry (`role.purpose`) and `{INDEX}` Purpose line (or the routing entry's description if no {INDEX} yet).
- Read the file's H1 plus first 1500 chars.
- If the file's topic clearly doesn't fit the folder's stated purpose → candidate. (The trigger is fast: keyword overlap < 0.2 between file content and folder Purpose.)

**Agent judgment:**
- Read both in full. Decide:
  - **Misplaced** → propose relocation to the right folder per the routing table.
  - **Folder purpose is too narrow** → propose broadening the {INDEX} Purpose to include the file's topic (route to the F9.2 fix).
  - **False positive** → the file is on-topic; the keyword trigger fired on incidental vocabulary. Drop it.

**Fix proposal:** target folder plus a redirect-wikilinks plan (any inbound `[[Old]]` references stay valid because wikilinks resolve by basename). Walk-only: the user confirms the target.

**Severity:** warn. Every warn ships a fix per the apply-everything rule.

**Fixable:** true. Walk-only.

## F9.5 — Folder-purpose duplication

**Framework rule:** no two folders should have substantially overlapping stated purposes. **Compares both standard and custom roles against each other** — `Notes/` (custom session) vs `Inbox/` (custom session) gets compared the same way `Resources/` vs `Library/` does.

**Trigger heuristic:**
- For every pair of folders in the registry (any layer except archive/meta): compute Purpose-line similarity (token overlap + entity overlap) using both `role.purpose` and the `{INDEX}` Purpose line.
- If similarity > 0.5 → candidate cluster.
- Also flag pairs where children sets overlap > 30% (the same files duplicated under different parents).

**Agent judgment:**
- Read both Purpose lines plus sample children. Decide:
  - **True duplication** → propose a merge (pick the canonical folder; route to the F9.6 reorg proposal).
  - **Adjacent but distinct** → propose Purpose-line clarification on both folders to make the distinction explicit (route to the F9.2 fix on each).
  - **False positive** → the folders genuinely cover different aspects; drop it.

**Fix proposal:** a merge plan (canonical folder + migration steps for children) OR purpose-clarification edits (drafted {INDEX} changes). Walk-only: the user confirms the direction.

**Severity:** warn → fix.

**Fixable:** true. Walk-only. Bigger merges automatically route to F9.6 (save-to-plan available).

## F9.6 — Architectural reorganization proposals

**Framework rule:** after F9.1–F9.5 run, surface the 1–3 highest-impact structural changes the vault would benefit from.

**Trigger heuristic:**
- Aggregate F9.1 stale routing + F9.4 misplacements + F9.5 duplications.
- Cluster proposals that touch the same folders.
- Rank by: (number of files affected) × (severity).
- Take the top 3.

**Agent judgment:**
- For each cluster, draft a migration plan: what moves where, what gets renamed, what gets merged, which {INDEX}s need rewriting. The reasoning **must cite the user's own stated purposes**: what the folders' own index files say they hold, what `roles.identity` / `roles.context` says about the user's work. The reasoning text must never justify the proposal with "the Brody Operating System does X" or any reference to a target taxonomy. Reorg proposals exist to resolve inconsistencies the user themselves implied (their stated folder purposes don't match contents; two folders they wrote purposes for are doing the same job; a file they wrote is unreachable from the root they wrote).
- If the cluster's underlying problem can be resolved by clarifying purposes rather than moving files, propose the clarification first: moving files carries the larger blast radius.

**Fix proposal — two modes (the user picks per proposal):**
- **Apply now (walk):** the agent walks each migration step, asks for confirmation per step, and executes immediately. High blast radius: pick this only if the user wants to commit in this run.
- **Save to plan:** the agent writes the full migration plan to `{save-to-plan path resolved via Step 4.3}/{YYYY-MM-DD}-reorg.md` with a checklist of steps. The user executes later (manually or via a follow-up optimizer run that consumes the plan).

Neither option is "skip." If the user picks save-to-plan, the plan file gets created: the proposal is committed to disk, not deferred indefinitely.

**Severity:** info (these are proposals, not violations). Per apply-everything: every proposal becomes either an applied migration or a saved plan.

**Fixable:** true. Walk-or-save-to-plan, user picks per proposal.

## F9.7 — Holistic CLAUDE.md fitness (vault-specific orientation)

**Framework rule:** the root `CLAUDE.md` must orient a fresh co-worker Claude in under 60 seconds toward this user's actual world.

**Trigger heuristic:**

1. **Read `roles.identity.path` and `roles.context.path` files** (cached in setup): build a 5-question orientation list specific to this user. Examples (the actual questions are generated from the discovered identity/context, not hardcoded):
   - "If the user asks about their current strategic focus, where does CLAUDE.md route me?"
   - "If the user mentions {project name found in identity/context}, can I find the project's authoritative file?"
   - "If a meeting note references {key person found in identity/context}, can I find their profile?"
   - "If I need to write in the user's voice, where's the brand/voice reference?"
   - "If I'm uncertain about a decision, where do I look for prior decisions on similar topics?"
   - If both identity and context are missing → use generic operator orientation questions and surface that gap simultaneously to F9.0.
2. **Read root `CLAUDE.md` cold** (a fresh read, no cached parsing). For each question, try to answer using only what CLAUDE.md says.
3. **Score each question:** answerable directly / answerable via routing-then-{INDEX} / not answerable.

**Agent judgment:**
- For each unanswerable or routing-fragile question, decide:
  - **Add a section to CLAUDE.md** that addresses the orientation gap (drafted by the agent).
  - **Improve a routing entry** so the answer is reachable in one hop.
  - **Add or strengthen a folder {INDEX}** if the answer exists but the index doesn't surface it.

**Fix proposal:** a specific edit to CLAUDE.md, routing, or {INDEX}. Walk-only: the user confirms per orientation gap. Edits to CLAUDE.md require explicit per-item approval (this preserves F1's auto-rewrite prohibition under user override).

**Severity:** fail. Orientation is foundational.

**Fixable:** true. Walk-only.

## Cross-framework constraints

| Constraint | Resolution in F9 |
|---|---|
| F1 forbids auto-rewriting CLAUDE.md | F9.1 / F9.7 rewrites of CLAUDE.md are walk-only with per-item approval. The user's "apply everything" rule applies after they confirm each draft. |
| F2.6 already covers routing presence + top-level mapping | F9.1 starts where F2.6 stops: it doesn't re-flag missing tables; it audits *truthfulness* of present tables. F2.6 findings + F9.1 findings are complementary, not duplicate. |
| F2.3 orphan-by-wikilink ≠ F9.3 orphan-by-navigation | Both run; both produce findings. A file can be a wikilink-orphan but navigation-reachable, or vice versa. Treat as distinct concerns. |
| F5 per-file budget vs F9.2 {INDEX} generation | {INDEX} format spec keeps files ≤ 8KB by construction. If a folder has so many children that {INDEX} would exceed 8KB → propose splitting the folder (route to F9.6) before generating {INDEX}. |
| F8.2 merge candidates vs F9.5 folder duplication | F8.2 operates on *files* across the curated layer (any role with `layer: curated`, including custom); F9.5 operates on *folders* across all non-archive non-meta roles. F9.5 proposals that resolve to file moves cite the corresponding F8.2 findings. |
| Custom roles | Are first-class. Every F9 check that operates on folders includes custom-role folders (provided their layer isn't archive/meta and the agent's confidence on the layer was at least medium). Low-confidence custom roles are gated behind an F9.0 clarification ask before they participate in F9.4 / F9.5 / F9.6. |
| G7.1 / G7.2 on generated content | {INDEX}, drafted CLAUDE.md sections, and reorg plan files all follow G7 conventions at write time. |
| {INDEX} is a navigation node, not a destination | F9.3 excludes {INDEX} from discoverability orphan checks. F2.3 includes it ({INDEX} should appear as a wikilink target from elsewhere — usually from the parent folder's {INDEX}). |

## Finding schema

```json
{
  "framework": "F9",
  "check_id": "F9.3",
  "check_name": "Discoverability walk",
  "path": "./Projects/positioning/research/competitor-X.md",
  "severity": "fail",
  "excerpt": "Navigation-orphaned: not listed in any {INDEX}; reachable only by direct file open.",
  "reasoning": "{roles.projects.path}/positioning/research/{INDEX} exists but only lists 3 of 7 files in the folder; competitor-X.md is one of the omitted files. The folder {INDEX} was last regenerated 2026-03-10; competitor-X.md was added 2026-04-22. The agent following root CLAUDE.md → routing entry → folder {INDEX} → child {INDEX} has no path to this file.",
  "action": "Add `[[competitor-X]] — analysis of Competitor X positioning` to Projects/positioning/research/{INDEX} under Children.",
  "proposed_target": "./Projects/positioning/research/{INDEX}",
  "proposed_edit": "## Children\n\n- [[existing-1]] — …\n- [[existing-2]] — …\n- [[existing-3]] — …\n- [[competitor-X]] — analysis of Competitor X positioning\n",
  "fix_modes": ["apply_now", "save_to_plan"],
  "fixable": true,
  "fixed": false,
  "citation": "anthropic-architecture.md → F9.3 Discoverability walk"
}
```

`reasoning` is mandatory and must explain the specific break in the navigation chain (which {INDEX}, when it drifted, what the agent's discovery path looks like with and without the fix).

`proposed_edit` is the literal text the fix would write. Walk-only confirmation: the user reviews this string before it's applied.

`fix_modes` is always `["apply_now", "save_to_plan"]` for F9.6 reorg proposals; `["apply_now"]` for the smaller F9.1–F9.5 / F9.7 fixes (which are individually low blast radius).
